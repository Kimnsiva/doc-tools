# KNOWN_ISSUES.md

Tracks accepted limitations and any bug that failed to fix after two attempts (per plan.md rule — logged and skipped rather than looped on).

## Phase 1

- **No PDF thumbnail preview in the tool view.** Section 6 of plan.md calls for first-page thumbnails; rendering those client-side needs a PDF rasterizer (e.g. pdf.js) which is a meaningfully larger addition than Phase 1's scope. Deferred — file rows currently show name + size only. Revisit when the frontend gains a build step or an offline-bundled pdf.js.
- **Compress endpoint's ≥30% size-reduction bar (section 8) is untested in Phase 1.** The Phase 1 test fixtures are blank vector-only pages (no embedded images), so ghostscript has little to compress. Real-world compression ratio testing needs an image-heavy PDF, which requires the images→PDF pipeline from Phase 2. TESTING.md notes this explicitly; re-verify once Phase 2 ships.
- **Repair's recovered page count is not guaranteed exact.** qpdf reconstructs a damaged file by scanning for objects when the xref/trailer is missing; how much is recoverable depends on where and how much of the file was corrupted. TESTING.md's expected page count is a "close to original" check, not an exact-match assertion — consistent with plan.md section 8's honesty bar for this feature.
- **Page reorder/rotate/crop tests rely on distinct page sizes, not visible text.** Blank test PDFs have no content stream to read back, so `scripts/generate_test_files.py` gives `sample_5page.pdf` a different page height per page and TESTING.md verifies ordering via `pdfinfo`'s "Page size" field instead of extracted text. This is a test-fixture design choice, not a product limitation.

## Phase 2

- **weasyprint 62.3 requires `pydyf==0.11.0`, not the latest 0.12.x.** Discovered during the Phase 2 Docker build verification: weasyprint 62.3's `pip install` metadata accepts `pydyf>=0.10.0`, but pydyf 0.12.x changed its `Stream` class API (`transform` moved), which crashes every `HTML(...).write_pdf(...)` call with `AttributeError: 'super' object has no attribute 'transform'`. Pinned `pydyf==0.11.0` in `backend/requirements.txt` with a comment explaining why — do not bump weasyprint or pydyf independently without re-testing `/api/html-to-pdf`.
- **`images-to-pdf`'s "original size" fit assumes 96 DPI.** img2pdf falls back to 96 DPI when an uploaded image has no embedded DPI metadata (true for the PNG/JPG fixtures the test generator produces), so a 300×400px image becomes a 225×300pt page, not "1 pixel = 1 point". This is img2pdf's documented default behavior, not a bug — TESTING.md states the expected math so it isn't mistaken for one.
- **`compress`'s ≥30% size-reduction bar (plan.md section 8) is still untested.** Phase 2 added the images↔PDF pipeline that could produce an image-heavy fixture, but no Phase 2 endpoint feeds back into `/api/compress`, so this is deferred again — revisit manually with a real scanned-looking PDF once convenient, or when Phase 3/OCR fixtures exist.

## Phase 1/2 doc fix (discovered while starting Phase 3)

- **TESTING.md's Phase 1 and Phase 2 sections were written against a stale, never-shipped API shape** (`-F "file=@x.pdf" $API/api/merge` style direct per-tool endpoints). The actual implemented backend only exposes `POST /api/files` (upload → `file_id`) and the generic `POST /api/ops/{tool}` / `GET /api/ops/{job_id}` (registry-driven, per plan.md's `TOOL_REGISTRY` note) — there never was an `/api/merge`, `/api/office-to-pdf`, etc. Rewrote every curl example in Phase 1/2 to the real upload→ops→download flow so the guide is actually copy-paste runnable. Also fixed `.env`/`.env.example`'s `FILE_TTL_MINUTES` (was `15`, drifted from plan.md's spec/`core/store.py`'s code default/the frontend footer copy, which all say `30`).

## Phase 3

- **ocrmypdf pinned to `16.10.4`, not the current latest (`17.x`/`16.13+`).** Those newer releases require `pikepdf>=10`, but `pikepdf==9.3.0` is pinned for the whole app (Phase 1/2/4-draft pikepdf code is written against it). `16.10.4` is the newest release still compatible with `pikepdf>=8.10.1`. Revisit together (bump pikepdf + re-test every pikepdf-based op) if a newer ocrmypdf is ever needed.
- **`camelot-py` pinned to `0.11.0` (pre-1.0 "classic" API), not the `1.x`/`2.x` maintainer-fork rewrite.** `ops/pdf_to_excel.py` uses the classic `camelot.read_pdf(...)` + `table.df` (pandas) surface; the 1.x/2.x API was not verified against this code. Installed `opencv-python-headless` + the `ghostscript` python binding directly instead of camelot's `[cv]` extra, which would otherwise pull in full `opencv-python` (needs `libGL`) alongside `pdf2docx`'s own `opencv-python-headless` dependency — two `cv2` installs can clash.
- **Japanese OCR/test-fixture font relies on `fonts-vlgothic` (single-file, no TTC index guessing), falling back to the merged `NotoSansCJK-Regular.ttc` at index 0** if `fonts-vlgothic` isn't found. `scripts/generate_test_files.py` globs for known font paths and prints a `WARNING: ... skipped` line instead of failing if neither is present (Debian font package layouts can shift between versions) — if `ocr_japanese.pdf` is missing after running the fixture script, check that warning first.
- **`scan_to_pdf`'s per-image cleanup (ImageMagick `convert` + `unpaper`) is best-effort.** If either subprocess fails on a given image, the code silently falls back to the original, uncleaned image for that page rather than failing the whole batch — consistent with plan.md's honesty bar (redact/compare aside, this tool doesn't promise cleanup will always improve OCR accuracy).
- **`pdf_to_excel`'s lattice→stream fallback only helps tables with at least some visible lines.** Fully borderless tables remain best-effort per plan.md section 9's honesty bar — `table_test.pdf` (a fully ruled table) is the fixture used to prove the happy path, not a stress test of the fallback.
- **OCR / Scan-to-PDF can only be tested inside Docker**, never directly on the Windows host — they need `tesseract`, its language packs, `unpaper`, and `imagemagick`, none of which this repo assumes are installed on the host machine.

## Phase 4

- **Scope grew beyond plan.md section 6's minimal Phase 4 list** (Protect/Unlock/Watermark/Page numbers/Sign/PDF-A). The backend (`backend/ops/*.py`) already had 5 extra tools drafted and registered from an earlier session — `change_permissions`, `add_stamp`, `flatten`, `sanitize`, `sign_certificate` — matching the broader iLovePDF-style "Security" category from `implementation_plan.md`. Since they were already fully implemented and registered in `TOOL_REGISTRY`, this phase wires all 11 into the frontend and TESTING.md rather than leaving 5 of them dead code; treat plan.md section 6's Phase 4 list as a floor, not a ceiling, going forward.
- **`sign` (image stamp) vs `sign_certificate` (real PAdES signature) are two different tools by design**, not a naming accident — `sign` is a visual rubber-stamp (see `core/stamping.py`), `sign_certificate` is a real digital signature via `pyhanko`. Both the UI copy and TESTING.md call out which is which so a user can't mistake the stamp for a legally-binding signature (plan.md's explicit warning for Phase 4).
- **`sign_certificate` is offline-only and best-effort**: no TSA/OCSP/CRL calls are made (consistent with plan.md's no-outbound-network-calls rule), so signatures have no trusted timestamp and no long-term validation (LTV) embedding. TESTING.md's self-signed test certificate will also always show as untrusted in any verifier — that's expected for a throwaway test cert, not a bug.
- **`pdfa`'s output is ghostscript's best-effort PDF/A conversion, not veraPDF-validated.** No veraPDF (or any PDF/A validator) is bundled, so TESTING.md's check is limited to "opens fine, page count unchanged" rather than strict spec conformance — consistent with the honesty bar in plan.md section 9.
- **`flatten` and `sanitize` have no fixture that actually exercises their real purpose.** No test PDF with form fields/annotations/embedded JS exists in `scripts/generate_test_files.py`, so TESTING.md can only prove `flatten` doesn't error on a plain PDF, and prove `sanitize` clears the `Producer`/`Creator` metadata pikepdf itself writes — neither test proves the tools work against adversarial/complex real-world input. Revisit if a real need for a richer fixture comes up.
- **Image/certificate uploads in the tool options panel are base64-JSON, not multipart.** `add_stamp`, `sign`, and `sign_certificate` need a second file (a stamp/signature image, or a `.p12` bundle) beyond the workspace's PDF, but the whole `POST /api/ops/{tool}` contract is JSON. The frontend (`workspace.js`) reads the chosen file client-side via `FileReader.readAsDataURL` and strips the `data:...;base64,` prefix before sending it as a normal JSON string field — no new upload endpoint was needed. This keeps the file-first workspace's single `file_id` model intact (the image/cert is a one-shot input to an op, never itself a workspace file).

## Phase 5

- **`compare` is a pixel diff, not a word-level diff** (plan.md section 9's stated limit). Pages are rendered at 100 dpi and a page is flagged when >0.5% of its pixels differ (gray deltas ≤32 ignored as antialiasing noise, `ops/compare.py` constants). Very small edits — a single changed character in small print — can fall under the threshold and go unflagged; conversely a re-export that shifts the whole layout by a few pixels flags every page. Both are inherent to image comparison, not bugs.
- **`compare` is capped at 100 pages per file** (both files fully rendered to images in memory is the cost driver). The error message tells the user to Split first. Also, when the two files' pages have different aspect ratios, file B's render is stretched to A's size before diffing — such pages almost always flag as changed, which is the sensible outcome anyway.
- **`compare`'s report labels files "A" and "B", not by filename.** Ops only receive file paths, not workspace metadata (`ToolSpec.run(paths, options, workdir)`), so the original filenames aren't available inside the op. A = first `file_id` sent (first checked in the UI), B = second; the report's summary page says so.
- **`redact` rasterizes the whole page, so ALL text on a redacted page becomes unsearchable**, not just the boxed region — visible text stays visible but `pdftotext`/copy-paste get nothing from that page (plan.md section 9: "redacted pages become raster", verified in TESTING.md). Untouched pages keep their text layer. Also expect larger file sizes (200-dpi PNG pages); chaining Compress afterwards is the workaround.
- **Redaction is only final once older versions are gone.** The workspace's undo keeps v1 recoverable until TTL expiry or manual file deletion — by design for every tool, but for redaction specifically TESTING.md warns: download the redacted version and remove the file from the workspace if the original must not linger.
- **Box-drawing precision is limited by the 200px-wide thumbnails.** Boxes are stored as 0–1 normalized page coordinates so they're resolution-independent, but a hand-drag on a 200px thumb has roughly ±0.5% jitter — fine for blacking out a line or paragraph, too coarse for redacting a single word in dense text. A zoomed page view would fix this; deferred (Phase 6 polish candidate at most).
